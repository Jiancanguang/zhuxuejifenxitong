## 二、学生成长追踪系统迁移计划

### 目标

将现有的单文件 HTML + IndexedDB 本地系统，迁移到已有的腾讯云服务器上，与学生积分系统共用后端和数据库，实现：

1. **数据上云**：不再依赖浏览器本地存储，支持多端同步
2. **统一入口**：教师在一个网站上同时管理积分系统和成长追踪
3. **家长端整合**：家长在同一个小程序中既能看积分也能看成长报告

### 迁移策略

推荐**渐进式迁移**，而非一次性重写：

```
阶段 1：后端 API + 数据库迁移（数据上云）
    ↓
阶段 2：教师端前端迁移（集成到现有 React 项目）
    ↓
阶段 3：家长端整合（小程序新增成长报告页面）
```

### 阶段 1：数据库与后端 API（预计 2-3 周）

**数据库设计**（新增表，不影响现有积分系统）：

```
growth_lessons        - 课程记录（对应原 lessons store）
growth_records        - 学生课堂记录（对应原 records store）
growth_exams          - 考试（对应原 exams store）
growth_exam_scores    - 考试成绩（对应原 exam_scores store）
growth_preps          - 备课记录（对应原 preps store）
growth_schedule_slots - 排课时间槽
growth_schedule_mappings - 排课班组映射
```

**需要做的事：**

- [ ] 设计 PostgreSQL 表结构（基于现有 IndexedDB 的 8 个 store）
- [ ] 编写数据库迁移脚本（migration）
- [ ] 开发后端 API 路由（CRUD）：
  - `/api/growth/lessons` - 课程记录
  - `/api/growth/records` - 学生课堂记录
  - `/api/growth/exams` - 考试管理
  - `/api/growth/exam-scores` - 考试成绩
  - `/api/growth/preps` - 备课记录
  - `/api/growth/schedule` - 排课管理
- [ ] 编写数据导入工具（将 IndexedDB 数据导出 JSON → 导入 PostgreSQL）
- [ ] API 鉴权复用现有教师登录系统

**关键决策：学生数据复用**

成长追踪系统的"学生"和积分系统的"学生"需要关联起来：
- 方案 A：共用同一个 students 表，给 students 表加字段（推荐，数据一致）
- 方案 B：成长系统独立学生表，通过映射关联（灵活但复杂）

建议选**方案 A**，在现有 students 表上扩展。

## 三、系统架构演进

### 当前架构

```
教师端（React）──→ Express API ──→ PostgreSQL
                       ↑
小程序（家长端）────────┘

成长追踪（独立 HTML）──→ IndexedDB（本地）
```

### 目标架构

```
教师端（React）
  ├── 积分系统模块
  ├── 成长追踪模块（新）
  └──→ Express API ──→ PostgreSQL
           ↑              ├── 积分相关表
小程序（家长端）            ├── 成长追踪表（新）
  ├── 积分查看              └── 家长账号表
  ├── 成长报告（新）
  └──→ API
```

## 四、技术注意事项

### 数据迁移风险

- IndexedDB 数据是浏览器本地的，需要先导出为 JSON
- 编写导入脚本时注意数据兼容（`normalizePrepRecord()`、`normMastery()` 等逻辑）
- 建议保留原 HTML 文件作为备用，迁移完成后再停用

### 共用基础设施

- 教师登录系统：复用现有 JWT 鉴权
- 学生数据：复用现有 students 表，扩展字段
- 家长账号：复用现有 parent_accounts 表
- 班级/班组：需要映射（积分系统的"班级"对应成长追踪的"班组"）

### 课后反馈表图片导出

- 现有方案用 html2canvas，React 版本可继续使用
- 小程序端需要用 Canvas API 重新实现（或后端生成图片）
